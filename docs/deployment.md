# Vercel deployment

Bandland supports Vercel only. Vercel Git integration owns Preview and Production deployments; no server bootstrap or deploy command is required.

## 1. Import and connect the project

Import the Git repository into Vercel as a Next.js project. Keep the framework defaults—`vercel.json` only adds project security headers.

Create two Blob stores from the project Storage page:

1. A **private** store for JSON content. In Advanced Options, set the environment-variable prefix to `CONTENT`, producing `CONTENT_BLOB_READ_WRITE_TOKEN`.
2. A **public** store for uploaded images. Set its prefix to `MEDIA`, producing `MEDIA_BLOB_READ_WRITE_TOKEN`.

Both stores are required in Production, Preview, and Development. Production and Preview read the `production` content namespace; local development reads `development`.

## 2. Configure environment variables

Set these in Vercel Project Settings:

- `CONTENT_BLOB_READ_WRITE_TOKEN` — added by the private Blob connection.
- `MEDIA_BLOB_READ_WRITE_TOKEN` — added by the public Blob connection.
- `ADMIN_PASSWORD_HASH` — bcrypt hash generated by `npm run auth:generate`.
- `AUTH_SECRET` — Auth.js secret generated by the same command.
- `NEXT_PUBLIC_SITE_URL` — canonical production URL, such as `https://example.com`.

Apply the canonical URL to every environment so Preview metadata remains canonical. Apply the auth values and both store tokens to every environment so authenticated Preview pages can read production data; the application rejects all Preview mutations and upload-token requests.

For local development:

```bash
npx vercel link
npx vercel env pull .env.local
npm run dev
```

Vercel-generated env files are gitignored. Never commit Blob tokens or authentication secrets.

## 3. Protect the admin login

In the Vercel Firewall dashboard, add a rate-limit rule with:

- Request method: `POST`
- Request path: `/api/auth/callback/credentials`
- Key: client IP
- Limit: 5 requests per 15 minutes
- Follow-up action: return HTTP 429

Publish the rule before directing the production domain to the project.

## 4. Deploy and verify

Push the target branch and allow Vercel to build it. Production starts with empty shows, merch, and audit data; sign in to `/admin` to create the first records.

Verify:

- `/`, `/shows`, and `/merch` render their empty states before content is added.
- Production admin create, edit, delete, and image upload flows work.
- Uploaded image URLs use the public Vercel Blob domain and are optimized by Next Image.
- A Preview deployment shows production content, displays its read-only banner, hides mutation controls, and rejects direct mutation/upload requests.
- The sixth rapid credentials request receives HTTP 429 from Vercel Firewall.

## Domains and rollback

Attach the production domain in Vercel Project Settings and keep `NEXT_PUBLIC_SITE_URL` aligned with it. Use Vercel's deployment history to promote or roll back application code. Content lives independently in Blob storage and is not changed by code rollbacks.

The private content store keeps the latest datasets and up to 100 audit entries. There is no hidden snapshot or media-archive subsystem; use the Vercel Blob dashboard for inspection and deliberate manual cleanup.
